The GUI's <code>api_config_get()</code> route correctly masks it (<code>"********"</code>) in GET responses, but <code>config.json</code> on disk is unprotected. Any process with filesystem read access can extract credentials. <strong>Immediate fix:</strong> store secrets in the <code>.env</code> file using <code>python-dotenv</code> and never serialize them to <code>config.json</code>.</p><p><strong>[HIGH] No authentication on the Flask web dashboard.</strong> <code>python main.py gui</code> starts a Flask server with zero authentication. While it defaults to <code>127.0.0.1</code> (localhost only), the README shows <code>--host</code> as a flag, meaning any user could bind it to <code>0.0.0.0</code> and expose contract management, alert configuration, and SMTP credentials to the network without any login requirement. <strong>Fix:</strong> add a simple <code>--password</code> flag that generates a session token and protects all routes with <code>@login_required</code>.</p><p><strong>[HIGH] <code>/api/config</code> POST endpoint with <code>_full_replace</code> flag allows arbitrary config replacement.</strong> The route <code>if data.get("_full_replace") or "contracts" in data: save_config(data)</code> will blindly overwrite the entire config with any JSON payload containing a <code>contracts</code> key. This is an SSRF/config-injection vector if the dashboard is ever exposed to the network.</p><p><strong>[HIGH] <code>urllib.request.urlopen()</code> in all webhook backends has no timeout.</strong> A hung Discord/Slack endpoint will block the executor thread indefinitely. <strong>Fix:</strong> add <code>timeout=10</code> to all <code>urlopen()</code> calls.</p><p><strong>[MEDIUM] <code>run_in_executor</code> with blocking <code>urlopen()</code> calls.</strong> The alert backends use <code>asyncio.get_event_loop().run_in_executor(None, self._send_sync, ...)</code> — the default executor uses a thread pool. With many contracts firing simultaneously, this could exhaust the thread pool. <strong>Fix:</strong> use <code>aiohttp</code> for all HTTP alert delivery.</p><p><strong>[MEDIUM] No input sanitization on contract address in URL path.</strong> The route <code>/api/contracts/&lt;address&gt;</code> passes the address directly to <code>remove_contract()</code>. While <code>remove_contract()</code> only does string comparison, future extensions could be vulnerable to path traversal if address values are used in file operations.</p><p><strong>[LOW] <code>webbrowser.open(url)</code> called before Flask server starts.</strong> In <code>cli.py</code>'s <code>gui()</code> command, <code>webbrowser.open(url)</code> is called immediately before <code>app_flask.run(...)</code>. There's a race condition where the browser opens before the server is ready. <strong>Fix:</strong> use a threading.Timer with a short delay.</p><hr><h3>1.5 Performance Issues</h3><p><strong>[HIGH] <code>_check_admin_calls()</code> fetches every full block with full transactions per polling cycle.</strong> For each block in the range <code>[prev+1, current]</code>, it calls <code>w3.eth.get_block(block_num, full_transactions=True)</code>. At a 15-second poll interval on Ethereum (~75 blocks per interval during catchup), this is 75 full RPC calls per contract per cycle. This is the most expensive operation in the codebase and will quickly exhaust free RPC tier limits. <strong>Fix:</strong> use <code>eth_getLogs</code> with function selector topics to replace the block-by-block scan.</p><p><strong>[MEDIUM] No RPC connection pooling or HTTP session reuse.</strong> Each <code>Web3.HTTPProvider(rpc)</code> creates a new HTTP connection object. At scale with many chains and contracts, connection overhead accumulates. <strong>Fix:</strong> pass a <code>requests.Session()</code> with connection pooling to the HTTPProvider.</p><p><strong>[LOW] <code>_token_decimals</code> cache is module-level global and never expires.</strong> This is fine for tokens (decimals rarely change) but if a token upgrades its contract, the cache will serve stale values indefinitely.
